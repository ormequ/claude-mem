# Адопт осиротевших ворктри: ревью и апрув через CLI

Статус: дизайн, к реализации.

## Проблема

Память ворктри становится видимой для запросов по родительскому проекту только
после адопта — проставления `merged_into_project`. Адопт запускается тремя
путями: проход на старте воркера, часовой `AdoptionScheduler` и ручной
`adopt-mem`. Все три перечисляют кандидатов из `git worktree list`
(`WorktreeAdoption.ts:179`).

Отсюда следует, что **удалённый ворктри не может быть заадоптен никогда**: его
нет в списке, значит он не рассматривается ни одним из путей. Строки остаются с
`merged_into_project IS NULL` навсегда.

Предполагалось, что до этого не дойдёт: `scripts/adopt-mem` задуман как хук на
удаление ворктри, и ради вызова из orca 11 июля создан симлинк
`~/.local/bin/adopt-mem`. Замер 2026-07-20: в `orca-data.json` поле
`automations` — пустой список. Хук не был подключён ни разу.

Накопившийся ущерб на 2026-07-20 — 8 проектов, 3886 строк:

| проект | наблюдений | саммари | последняя запись |
|---|---|---|---|
| `kedo/dropdown-primevue` | 173 | 29 | 2026-07-20 |
| `kedo/frontend-toolchain-host` | 25 | 1 | 2026-07-20 |
| `kedo/modal-header-refactor` | 367 | 83 | 2026-07-11 |
| `kedo/octane-permissions-teamid-reset` | 166 | 40 | 2026-07-06 |
| `kedo/refactor-block-b2` | 6 | 1 | 2026-06-29 |
| `kedo/refactor-foundation` | 1879 | 240 | 2026-06-28 |
| `kedo/refactor-block-table` | 673 | 55 | 2026-06-28 |

Два последних дропнуты в течение суток, второй — прямо во время сессии, в
которой писалась эта спека.

### Что НЕ является проблемой

Доказательство мержа. При удалении ворктри claude-mem вопрос «влито или
брошено» **сознательно не задаёт**: путь `--branch` пропускает проверку на мерж
и адоптит безусловно (`WorktreeAdoption.ts:188-190`). Проверка `--merged` +
`isSquashMerged` работает только в режиме сканирования всех ворктри. Поэтому
CLI ничего не доказывает — он показывает факты, решение принимает человек.

## Контекст: два поколения ворктри

Пользователь переиспользует ворктри — инициализация проекта тяжёлая (Docker,
зависимости, БД). `create_worktree` даёт каталогу случайное имя
(`amber-harbor`, `crisp-willow`), и это имя служит **стабильной идентичностью
контейнера** для claude-mem; ветки внутри меняются через `git switch -c`.

Следствия, которые обязан учитывать дизайн:

1. **`--branch` — селектор ворктри, а не фильтр строк.** UPDATE идёт по
   `WHERE project = <имя-каталога>` (`WorktreeAdoption.ts:246-249`), ветка в
   условие не входит. В переиспользуемом ворктри адопт забирает память всех
   веток, которые там жили. Для контейнерной идентичности это верно, но по
   имени флага читается наоборот.
2. **Живой ворктри с неадопченной памятью — норма, а не дефект.** Для
   переиспользуемого контейнера `adopt-mem` запускается по ходу жизни, без
   удаления. Такой ворктри копит память по нескольким фичам сразу, и человеку
   полезно видеть, когда пора.

## Решение: `adopt-mem --orphans`

Интерактивный обзор с апрувом. Ничего не адоптит без явного ответа.

### Детекция

1. Из БД взять `project` со `merged_into_project IS NULL`, содержащие `/`.
   Родитель — часть до первого `/`. Composite строится как
   `basename(parent)/basename(cwd)` (`worktree.ts:57-59`), слэш внутри
   basename невозможен, поэтому разбор по первому `/` корректен.
2. Из `known-cwds.json` резолвить каждый cwd в корень репы
   (`git rev-parse --path-format=absolute --git-common-dir`) и построить
   отображение «имя родительского проекта → **множество** путей реп».
   Именно множество, а не один путь: имя проекта — это basename корня репы,
   и две репы с одинаковым basename (второй клон, бэкап) коллидируют. Если
   победит не та, живые ворктри другой репы не найдутся в её списке и будут
   объявлены сиротами. Сирота = отсутствует во **всех** путях этого имени;
   при неоднозначности печатать предупреждение.
3. Для каждой известной репы получить ворктри (`git worktree list
   --porcelain`) и их имена проектов через `getProjectContext(path).primary`.
4. **Правило живости (обязательное).** Запись из `git worktree list` считается
   живой, только если её каталог существует на диске и у неё нет атрибута
   `prunable`. Git показывает удалённый через `rm -rf` ворктри до `git worktree
   prune` — с пометкой `prunable gitdir file points to non-existent location`.
   Ручное удаление и есть основной сценарий этого тула, так что без явной
   проверки такая сирота не всплывёт никогда.

   Сейчас корректность держится на случайности: `findGitRepoRoot`
   (`project-name.ts:21-34`) на несуществующем каталоге уходит в `catch` и
   возвращает `null`, из-за чего имя выходит без родительского префикса и не
   совпадает с записью в БД. Реализация, которая соберёт composite вручную
   (`${parent}/${basename(path)}`), сломает классификацию молча. Проверять
   существование каталога и `prunable` явно, не полагаясь на fallback.
5. Классификация:
   - каталога нет среди живых → **сирота**, идёт в апрув;
   - есть среди живых → **живой**, идёт в информационную секцию;
   - родитель не сматчился ни с одной известной репой → **пропустить** с
     предупреждением. Подтвердить, что ворктри мёртв, нечем, а молчаливый
     показ его как сироты приведёт к ошибочному адопту.

   Пропуск тупиковый: `known-cwds.json` — единственный источник резолва (ни
   `observations`, ни `sdk_sessions` пути не хранят), он вычищается на запись,
   при порче обнуляется и имеет потолок `MAX_CWDS = 500`. Если у репы в реестре
   остались только удалённые worktree-пути, её сирот не заадоптить ничем:
   обычный `adopt` требует живого ворктри. Поэтому нужен запасной выход —
   флаг `--repo <path>`, задающий корень репы вручную.

### Вывод

```
Осиротевшие ворктри (каталог удалён) — kedo
  1. kedo/dropdown-primevue          202 строки   20 июл – 20 июл
  2. kedo/frontend-toolchain-host     26 строк    20 июл – 20 июл
  ...
Заадоптить kedo/dropdown-primevue в kedo? [y/n/a/q]

Живые ворктри с неадопченной памятью (не трогаю):
  kedo/crisp-willow      235 строк    запусти adopt-mem <путь>, когда закончишь
```

Ответы: `y` — заадоптить, `n` — отказ (запоминается), `a` — все оставшиеся
сироты, `q` — выход без изменений для оставшихся.

Метрика везде одна — `наблюдения + саммари`. Смешивать нельзя: в черновике
этой спеки сироты считались суммой, а живые — только наблюдениями, что давало
несопоставимые числа в соседних секциях.

**Риск ответа `a`.** Хук удаления срабатывает в момент, когда человек помнит
статус ветки; `--orphans` спрашивает недели спустя (`refactor-foundation` —
три недели). `a` в сочетании с выцветшей памятью — основной источник
ошибочных адоптов. Поэтому даты в выводе обязательны, а `a` применяется
только к оставшимся в текущем прогоне, никогда не к будущим.

### Отказы

`~/.claude-mem/state/orphan-decisions.json`, по образцу `known-cwds.json`:

```json
{ "declined": [{ "project": "kedo/refactor-block-b2", "at": "2026-07-21T…" }],
  "updatedAt": "…" }
```

Отказанные не показываются в секции апрува при следующих запусках. Снимается
флагом `--reset-declined`.

**Отказ хранится с меткой времени, и это обязательно.** Имена ворктри у
пользователя не только случайные — есть осмысленные (`api-reanimation`,
`dropdown-primevue`), и такое имя может быть переиспользовано. Отказ по голому
имени навсегда скрыл бы память НОВОГО ворктри с тем же именем. Правило: проект
показывается снова, если у него появились строки новее отказа
(`MAX(created_at) > declined.at`).

Тот же механизм покрывает лаг компрессии: строки, дописанные воркером уже
после адопта, снова покажут проект сиротой на следующем прогоне. Это
ожидаемое поведение, а не дефект.

Снимка состояния улик не хранится: доказательство мержа из схемы исключено.

### Сам адопт

Тот же UPDATE, что в `WorktreeAdoption.ts:238-248`, в транзакции на проект:

```sql
UPDATE observations      SET merged_into_project = ?, chroma_merge_synced_at = NULL
  WHERE project = ? AND merged_into_project IS NULL;
UPDATE session_summaries SET merged_into_project = ?, chroma_merge_synced_at = NULL
  WHERE project = ? AND merged_into_project IS NULL;
```

Условие `merged_into_project IS NULL` делает операцию идемпотентной. Сброс
`chroma_merge_synced_at` ставит строки в очередь `ChromaMergeDrain`; Chroma
патчит воркер на следующем дрейне. **Свою писалку в Chroma не открывать** —
вторая писалка по тому же стору это инцидент 2026-07-11, см.
`FORK_NOTES.md:313-338`.

### Флаги

- `--orphans` — интерактивный обзор;
- `--orphans --dry-run` — только список, без промптов и записи;
- `--orphans --reset-declined` — очистить список отказов;
- `--orphans --repo <path>` — задать корень репы вручную, когда резолв через
  `known-cwds.json` не удался;
- нет TTY → печатает список и выходит с кодом 0, ничего не адоптит.

В выводе после адопта явно сказать, что Chroma обновится на следующем дрейне
воркера (до часа) — иначе отсутствие изменений в семантическом поиске сразу
после команды читается как сбой.

## Размещение кода

Дельта к апстриму — **ноль строк**. Ограничение осознанное: `WorktreeAdoption.ts`
апстримный и уже несёт форк-дельту в 112 строк, углублять её дорого при
следующем мерже (тот же аргумент, что в `FORK_NOTES.md:114-120`).

- `src/services/infrastructure/OrphanAdoption.ts` — новый, форк-овный: детекция,
  классификация, адопт, хранилище отказов. Рядом с `AdoptionScheduler.ts` и
  `KnownCwdRegistry.ts`, которые тоже форк-овные.
- `scripts/adopt-orphans.ts` — новый: TTY-слой, промпты, форматирование.
- `scripts/adopt-mem` — правка: ветка `--orphans` делегирует в него.

Git-хелперы (`gitCapture`, `listWorktrees`) дублируются из `WorktreeAdoption.ts`
вместо экспорта: дублирование ~30 строк в своём файле дешевле владения
апстримной функцией. Копия `listWorktrees` обязана разбирать атрибут
`prunable` — упомянутый выше оригинал этого не делает.

**Откуда запускается.** `adopt-mem` исполняет УСТАНОВЛЕННЫЙ билд из
plugin-cache (`worker-service.cjs`), а `adopt-orphans.ts` будет импортировать
`src/` dev-чекаута. Это разные срезы кода, и при расхождении схемы между
установленным плагином и исходниками возможен version skew. Команда
предназначена для запуска из dev-чекаута; на схему она полагается только в
части колонок `merged_into_project` и `chroma_merge_synced_at` — их наличие
проверять через `PRAGMA table_info`, как это делает
`WorktreeAdoption.ts:213-223`, и выходить с внятной ошибкой, а не падать.

## Границы: что НЕ входит

- **Бэкстоп в `KnownCwdRegistry`** (адопт в момент вычистки мёртвого пути).
  Отложено: archive-хук закрывает удаление через orca, а `git worktree remove`
  руками остаётся под этим CLI.
- **Запись улик** (ветка + SHA живых ворктри). Отпадает: после починки
  archive-хука будущие дропы адоптятся в момент удаления, копить нечего.
- **Доказательство мержа** — см. «Что НЕ является проблемой».
- **Починка archive-скрипта в orca** — конфигурация, не код; делается отдельно
  и является основным исправлением. Этот CLI — разбор накопленного и страховка.
- **Апрув живых ворктри** — только информационная секция.

## Тесты

`tests/services/infrastructure/orphan-adoption.test.ts`:

1. проект без живого каталога → сирота; с живым → живой;
2. **запись `git worktree list` с несуществующим каталогом / атрибутом
   `prunable` → НЕ живой** (сценарий `rm -rf` без `git worktree remove`);
3. **две репы с одинаковым basename → ворктри второй не объявлен сиротой**
   (коллизия имён родителей);
4. родитель без сматченной репы → пропущен, не помечен сиротой;
5. `--repo <path>` резолвит родителя, когда реестр не помог;
6. отказанный проект отсутствует в списке на апрув, но остаётся в БД;
7. **отказанный проект появляется снова, если у него есть строки новее отказа**;
8. `--reset-declined` возвращает его в список;
9. битый `orphan-decisions.json` → пустой список отказов (fail-open: спросить
   лишний раз безопаснее, чем скрыть);
10. адопт проставляет `merged_into_project` и обнуляет `chroma_merge_synced_at`
    и в `observations`, и в `session_summaries`;
11. повторный адопт меняет 0 строк (идемпотентность);
12. родитель выводится из строки проекта: `kedo/x` → `kedo`;
13. `--dry-run` и отсутствие TTY не пишут ни в БД, ни в файл отказов.

Прогонять с отключённой песочницей — она запрещает чтение `./node_modules`
(`FORK_NOTES.md:360-363`).
